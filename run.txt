Frontier-Eng 运行说明（run.txt）
================================

【零、环境已配好后：跑 v1「全量」评测用哪些命令】
------------------------------------------------
说明（必读）：

  • 仓库 **没有** 单个 `--matrix` 覆盖全部 v1 任务；官方按机器角色拆成多个 YAML（CPU / GPU /
    FlashAttention / EngDesign）。四份都跑完，才覆盖当前 `frontier_eval/conf/batch/` 里维护的
    v1 批量任务集合（与 leaderboard 的拆分方式一致；个别任务如 MuonTomography 可能未进 batch，
    见 frontier_eval/README.md「v1 Task Environments」小节）。

  • 前置：已执行 `bash init.sh` 且 `conda activate frontier-eval-2`；已按需执行
    `bash scripts/setup_v1_merged_task_envs.sh`；根目录 `.env` 已配置可用的 `OPENAI_API_KEY`
   （及网关则设 `OPENAI_API_BASE`）。下列 matrix 里多为 **algorithm.iterations=100**，
    **耗时长、会产生大量 API 费用**。

  • EngDesign 批需要 Docker 相关变量（与 matrix 文件头一致）；GPU 批需本机 CUDA 与
    `frontier-v1-kernel` 等环境已按 README 就绪。

在仓库根目录依次执行（Bash / WSL；上一段跑完再跑下一段）：

     export PYTHONNOUSERSITE=1
     conda activate frontier-eval-2

     python -m frontier_eval.batch --matrix frontier_eval/conf/batch/v1_cpu_openevolve_p8_i100_gemini-3.1-pro-preview.yaml

     CUDA_VISIBLE_DEVICES=0 python -m frontier_eval.batch --matrix frontier_eval/conf/batch/v1_gpu_openevolve_qwen3codernext_100.yaml

     CUDA_VISIBLE_DEVICES=0 python -m frontier_eval.batch --matrix frontier_eval/conf/batch/v1_flashattention_openevolve_qwen3codernext_100.yaml

     export ENGDESIGN_EVAL_MODE=docker
     export ENGDESIGN_DOCKER_IMAGE=engdesign-sim
     python -m frontier_eval.batch --matrix frontier_eval/conf/batch/v1_engdesign_openevolve_qwen3codernext_100.yaml

Windows PowerShell 等价写法示例（按需改显卡号、路径保持为仓库根目录）：

     $env:PYTHONNOUSERSITE = "1"
     conda activate frontier-eval-2

     python -m frontier_eval.batch --matrix frontier_eval/conf/batch/v1_cpu_openevolve_p8_i100_gemini-3.1-pro-preview.yaml

     $env:CUDA_VISIBLE_DEVICES = "0"
     python -m frontier_eval.batch --matrix frontier_eval/conf/batch/v1_gpu_openevolve_qwen3codernext_100.yaml
     python -m frontier_eval.batch --matrix frontier_eval/conf/batch/v1_flashattention_openevolve_qwen3codernext_100.yaml

     $env:ENGDESIGN_EVAL_MODE = "docker"
     $env:ENGDESIGN_DOCKER_IMAGE = "engdesign-sim"
     python -m frontier_eval.batch --matrix frontier_eval/conf/batch/v1_engdesign_openevolve_qwen3codernext_100.yaml

输出：每段在 `runs/batch/<run.name>/` 下生成独立结果与 `summary.jsonl`。若仅需 **验证合并环境
是否可用（iter=0）** 而非全量演化，见仓库 `scripts/validate_v1_merged_task_envs.sh`。


本文区分：
  • 「冒烟」：几乎不跑演化，只验证调度/链路（algorithm.iterations=0 等）。
  • 「完整跑」：对真实任务做多轮演化/搜索（iterations 或等价参数 > 0），且任务子进程能真正执行校验。

更细的 Hydra 参数、算法差异、任务覆盖项以 frontier_eval/README.md 为准。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
一、共用前置：驱动环境 + 隔离
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1) 仓库根目录执行（Git Bash / WSL；需已安装 conda）：

     bash init.sh
     conda activate frontier-eval-2

2) 完整跑之前建议始终设置（与仓库 README 一致，避免用户站点包污染任务隔离）：

     export PYTHONNOUSERSITE=1

   Windows PowerShell 示例：

     $env:PYTHONNOUSERSITE = "1"
     $env:PYTHONUTF8 = "1"
     $env:PYTHONIOENCODING = "utf-8"

3) 演化需要 LLM：复制并编辑 .env（init.sh 已可能帮你生成）：

     # 至少配置 OPENAI_API_KEY；兼容网关可设 OPENAI_API_BASE
     # 见 .env.example

   运行 frontier_eval 时会从当前目录向上查找并加载最近的 .env。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
二、冒烟（快速验证驱动，不是完整评测）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  • 框架自带 smoke 任务（不依赖外部 benchmark 体积依赖时可用）：

     python -m frontier_eval task=smoke algorithm=openevolve algorithm.iterations=0

  • 默认 unified + 某 benchmark、0 次迭代（仍可能因该任务缺 Octave/runtime 而子进程失败）：

     python -m frontier_eval algorithm.iterations=0


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
三、完整跑起来（非冒烟）—— 你需要多做的事
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

完整跑 = 驱动环境（frontier-eval-2）+ 该任务对应的 runtime 环境/依赖 + 有效的 .env + iterations>0。

A) 安装一批合并好的任务 runtime（推荐先做，耗时长、占磁盘）

     bash scripts/setup_v1_merged_task_envs.sh

   装好后，任务会通过 conda 环境名或 python_path 指向例如 frontier-v1-main、frontier-v1-kernel 等
   （具体以各 benchmark 的 YAML/README 为准）。

B) 按任务补全「本地说明」里的要求（无法在一页写全 47 个任务）：

   运行前必读：

     benchmarks/<Domain>/README*.md
     benchmarks/<Domain>/<Task>/README*.md（若有）

   按说明安装：额外 pip、Docker、GPU 环境、third_party 克隆、数据下载等。
   仓库根 README 的 Getting Started 也列出了常见坑（DuckDB/EV2Gym、Optics、MolecularMechanics、
   GPU kernel、外部资产 dc-rl / PhySense / SustainDC 等）。

C) 算法额外依赖（仅当你用对应算法时需要）

   • openevolve：随 frontier_eval/requirements.txt，一般驱动环境即可。
   • shinkaevolve：需按 frontier_eval/README.md 安装 ShinkaEvolve（含 third_party 与 patch 等）。
   • abmcts：需 third_party/treequest 等，见 frontier_eval/README.md。

D) 完整跑示例命令（OpenEvolve + unified + 指定任务 + 多轮迭代）

   将 <Domain>/<Task> 换成你要跑的任务，例如 ComputerSystems/MallocLab：

     python -m frontier_eval ^
       task=unified ^
       task.benchmark=ComputerSystems/MallocLab ^
       algorithm=openevolve ^
       algorithm.iterations=10

   Linux/macOS 把 ^ 换成行末反斜杠 \。

   其他示例（EngDesign 等）见 frontier_eval/README.md 的「Unified task」「EngDesign」小节。

E) 指定模型（在命令行或配置里覆盖 llm.*）

   例如：

     python -m frontier_eval task=unified task.benchmark=... algorithm=openevolve algorithm.iterations=10 llm.model=gpt-4o-mini

   具体模型名需与你的 API / 网关一致。

F) 批量矩阵（多组任务/模型一次跑）

     python -m frontier_eval.batch --matrix frontier_eval/conf/batch/example_matrix.yaml

   复制 example 矩阵改自己的 sweep；路径与写法见 frontier_eval/README.md。

G) 最小「完整演化」闭环（验证 API + OpenEvolve + iterations>0，不拉真实 benchmark 重依赖）

   用途：在不动 47 个工程任务的前提下，先确认「配好密钥后，演化能多轮跑完」。
   smoke 任务只在本机用当前 Python 执行候选脚本（见 frontier_eval/tasks/smoke/），
   不依赖 Octave、合并 runtime 等，是最轻的完整演化路径。

   前置（必须）：

     cp .env.example .env

   编辑 .env，至少填写 OPENAI_API_KEY；若用自建兼容网关，再填 OPENAI_API_BASE。

   示例命令（将迭代数按需改大，如 10、50；会产生 LLM 费用）：

     python -m frontier_eval task=smoke algorithm=openevolve algorithm.iterations=3

   Linux/macOS 可写成一行；Windows CMD 同理。

   本机实测记录（供对照）：

     • 未放置 .env、或 OPENAI_API_KEY 为空时，执行
       algorithm.iterations=1 会在启动阶段失败，提示：
       Missing API key for OpenEvolve. Set OPENAI_API_KEY or llm.api_key
       when algorithm.iterations > 0.
     • 配好密钥后，应进入正常演化循环；若仍失败，检查网络、代理与 llm.model 是否与网关一致。

四、怎样算「完整跑通」某个任务

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  • 冒烟：进程退出码 0 只说明调度/框架侧 OK；任务校验仍可能失败。
  • 完整跑：对该任务在 iterations>0 下能持续调用 evaluator、基准 returncode 与分数符合预期，
    且已按该任务 README 处理 runtime、资源与权限。

若仍卡在 Windows 上子进程 9009 等，优先检查：该任务是否依赖 Octave/Unix 脚本、PATH、
以及是否应改用 WSL/Linux 或与任务文档一致的 conda 环境名。
